# PRD: 워런티 용어 통일 + 프로젝트 메인 일정 마일스톤 연동 (2026-02-15)

## 배경
현재 UI에서 `워런티`, `AS`, `유지보수` 용어가 혼용되어 사용자가 프로젝트 타입/단계 의미를 직관적으로 이해하기 어렵다. 또한 프로젝트 메인 페이지의 일정 마일스톤(타임라인)은 실제 일정관리(WBS) 데이터와 연동되지 않아 신뢰도가 낮다.

## 목표
1. UI 전반의 `AS/유지보수` 표기를 **워런티**로 통일한다.
2. 프로젝트 메인 페이지의 일정 마일스톤을 **설계-제작-설치-워런티** 4단계로 표시하고, 일정관리(WBS) 입력값을 반영한다.
3. 프로젝트가 `검토/종료` 단계인 경우, 마일스톤 영역 위에 **크게 오버레이로 검토/종료**를 표시한다.
4. 마일스톤 영역의 기존 디자인(레이아웃/톤/컴포넌트)은 유지한다.

## 비목표
- 일정관리(WBS) 데이터 구조/시스템 그룹을 변경하지 않는다.
- 워런티를 위한 별도 WBS 시스템 그룹을 추가하지 않는다.

## 요구사항
### FE-REQ
- FE-REQ-001: 프로젝트 타입/필터/라벨에서 `AS`, `유지보수` 표기를 **워런티**로 통일한다.
- FE-REQ-002: 프로젝트 메인 페이지의 일정 마일스톤은 `설계-제작-설치-워런티` 4단계로 표시한다.
- FE-REQ-003: 프로젝트 메인 페이지의 마일스톤 날짜는 일정관리(`GET /budget/projects/{project_id}/schedule`)의 WBS 데이터를 기반으로 계산한다.
  - 설계/제작/설치: 각 단계 root 그룹(`stage-design`, `stage-fabrication`, `stage-installation`)의 `first_start ~ last_end` 범위를 표시한다.
  - 워런티: `설치`의 `last_end`를 워런티 시작일로 사용하고, 종료일은 시작일의 1년 후 날짜로 표시한다.
- FE-REQ-004: 프로젝트가 `검토/종료` 단계이면 마일스톤 위에 오버레이로 **크게 검토/종료**를 표시한다(반투명 글래스 스타일).
- FE-REQ-005: 일정 데이터가 로딩 중이거나 비어있는 경우 UI는 깨지지 않아야 하며, 플레이스홀더(`...` 또는 `-`)로 표시한다.

### BE-REQ
- BE-REQ-001: 프로젝트 타입 코드 `as`의 표시 라벨은 API 응답에서 **워런티**로 직렬화한다.
- BE-REQ-002: 사용자에게 노출되는 메시지에서 `AS` 표기는 **워런티**로 변경한다.

## 수용 기준(AC)
1. 주요 UI에서 `AS/유지보수` 표기가 `워런티`로 통일되어 표시된다.
2. 프로젝트 메인 페이지의 일정 마일스톤이 4단계(설계/제작/설치/워런티)로 표시된다.
3. 일정관리에서 입력한 설계/제작/설치 일정이 마일스톤 날짜에 반영된다.
4. 설치 종료일을 기준으로 워런티 시작/종료(1년 후)가 계산되어 표시된다.
5. 프로젝트가 `검토/종료` 단계일 때 마일스톤 위에 큰 오버레이가 표시된다.
6. `docker exec -w /app synchub_web bash scripts/verify_fast.sh`가 통과한다.
7. `docker exec -w /app synchub_frontend npm run build`가 통과한다.
