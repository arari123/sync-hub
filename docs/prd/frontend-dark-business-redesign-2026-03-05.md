# PRD: 프론트엔드 전면 점검 및 다크 비즈니스 디자인 개편 (2026-03-05)

## 1) 배경
- 현재 프론트엔드는 페이지별 스타일 편차가 크고, `slate/white` 중심의 라이트 팔레트가 다수 잔존해 다크 환경에서 일관성이 떨어진다.
- 상단 퀵메뉴/검색 헤더 등 공통 UI가 일부 페이지에서 중복 구현되어 유지보수 비용이 높다.

## 2) 목표
- 전문적인 비즈니스 웹 톤의 다크모드 UI를 기본 제공한다.
- 공통 레이아웃/토큰/상단 네비게이션을 중심으로 스타일 일관성과 재사용성을 높인다.
- 핵심 사용 경로(홈/검색/문서상세/안건 스플릿뷰/인증 화면)의 시각 품질과 가독성을 개선한다.

## 3) 범위
### 포함
- 전역 테마 토큰 및 바탕 배경/서피스/타이포 재정의
- 공통 UI 프리미티브(`Button`, `Input`, `Logo`, `Layout`, `GlobalTopBar`) 리팩토링
- 상단 퀵메뉴 공통 컴포넌트 분리(중복 제거)
- 핵심 화면 다크 개편
  - `/home` (`SearchResults`)
  - 문서 결과/상세 컴포넌트
  - 안건 Split View 및 에디터
  - 인증(로그인/회원가입/이메일 인증)
- 레거시 색상 유틸 클래스(`white/slate`) 다크 호환 오버라이드 레이어 적용

### 제외
- 백엔드 API 변경
- IA/라우팅 구조 변경
- 신규 기능(비즈니스 로직) 추가

## 4) 상세 요구사항
- REQ-001: 앱 기본 렌더링이 다크모드로 시작되어야 한다.
- REQ-002: 전역 스타일 토큰이 다크 중심으로 정비되고, 카드/테두리/텍스트 대비가 확보되어야 한다.
- REQ-003: 상단 퀵메뉴 UI는 재사용 컴포넌트로 통합되어 중복 구현을 제거해야 한다.
- REQ-004: `SearchResults` 상단/필터/카드/페이징 영역은 전문적 비즈니스 톤으로 개편되어야 한다.
- REQ-005: `ResultList`, `GlobalSearchResultList`, `DocumentDetail`, `AgendaSplitView`, `RichTextEditor`는 다크 배경에서 가독성이 유지되어야 한다.
- REQ-006: 인증 화면(`Login`, `Signup`, `VerifyEmail`)은 다크 톤으로 일관화되어야 한다.
- REQ-007: Docker 환경에서 `verify:fast` 및 프론트 집중 ESLint가 통과해야 한다.

## 5) 수용 기준
- 핵심 페이지에서 배경/서피스/텍스트/상태색이 라이트 톤으로 깨지지 않는다.
- 공통 UI 변경으로 빌드/린트 오류가 발생하지 않는다.
- 검증 명령 통과 로그를 남긴다.

## 6) 리스크 및 대응
- 리스크: 대형 페이지(`SearchResults`) 내 하드코딩 클래스가 많아 누락 가능성 존재
- 대응: 전역 오버라이드 + 핵심 컴포넌트 직접 수정을 병행해 화면 누락을 완화
